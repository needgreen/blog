# LXP 프로젝트 - 7일간의 협업 여정

![image.png](./images/image.png)

## 프로젝트에 들어가며

“과연 일주일 만에 완성도 있는 강의 플랫폼을 만들 수 있을까?”

시작 전에는 걱정이 앞섰지만 다행히 프로젝트에 들어가기 전, 자바스크립트만으로 도서 관리 앱을 만드는 실습을 통해 기초를 다시 한번 다졌고 추석 연휴 동안 복습할 시간도 가질 수 있었다.

또한 팀 구성을 미리 공유받아 본격적인 시작 전 구조를 구상하고 회의를 진행할 수 있었던 점도 큰 도움이 되었다. 이런 준비 덕분에 세 명의 동료가 모여 ‘협업의 힘’을 만들 수 있었다.

프로젝트의 목표는 **HTML, CSS, JavaScript만으로 실제 서비스 수준의 온라인 강의 플랫폼을 구현하는 것**이었다. 1차 프로젝트는 본격적인 2차 프로젝트로 넘어가기 전, 워밍업의 성격이 강했다. 그래서 완벽한 결과물보다는 **자바스크립트의 기능 구현에 대한 이해를 높이는 것**에 중점을 두었다.

우리는 프레임워크 없이 순수 JavaScript로 강의 등록부터 조회, 수정, 삭제까지 구현하는 데 도전했다.

### 기술 스택

- **Frontend**: Vanilla JavaScript, HTML, CSS
- **Storage**: localStorage API
- **Version Control**: Git Flow
- **Collaboration**: Notion, Slack, Figma

## 프로젝트 준비 과정

### 1단계 : MVP 기능 정의 (10.7)

첫 번째 단계는 **무엇을 만들 것인지 명확히 하는 것** 이었다. 일정 내 구현 가능한 핵심 기능을 정의하고, 여유가 생기면 추가할 기능을 리스트업하고 개발 흐름을 체계적으로 관리하기 위해 주요 **마일스톤**을 함께 계획했다.

- **MVP 기능**
  - 강좌 목록 조회 (카드형 UI)
  - 필터링 (카테고리, 난이도)
  - 페이지네이션 (8개씩 표시)
  - 강좌 등록 (제목, 썸네일, 카테고리, 난이도 등)
  - 강좌 상세보기
  - 정렬 (최신순/평점순/수강인원순)
- **추가 기능**
  - 강좌 수정/삭제
  - 검색 기능 (제목/강사/태그 기반)

> 💡 **배운 점 :** MVP를 먼저 정의하니 개발 우선순위가 명확해졌고, QA 일정을 두 번으로 나누어 중간 점검을 진행함으로써 현실적인 일정 관리가 가능했다.

### 2단계 : 데이터 구조 설계

강의 플랫폼의 핵심은 **데이터 구조 설계**였다. 2회에 걸친 온라인 회의를 통해 각자 구상한 데이터를 공유하고 이를 바탕으로 **확장 가능한 최종 데이터 구조**를 도출했다.

```jsx
const courseList = {
	id: '11',
	title: '',
	description: '', // 요약
	content: '' // 상세설명
	thumbnailUrl: '',
  instructor: '김코딩',
	category: ['프로그래밍', '웹', '프론트엔드'], // 프로그래밍 > 웹 > 프론트엔드 ['디자인', 'UI/UX']
  level: "초급", // 입문, 초급, 중급, 상급
  tags: ['프론트엔드', '입문'], // 보여주기 위한 데이터 요소 ([category[category.length-1], level])
  reviews: [{
	  rating:
  },],
  studentCount: 10,
  createdAt: "2025-10-10T00:00:00Z"
  }
```

> 💡 **배운 점 :** 데이터 구조를 먼저 확정하니 동료 간 커뮤니케이션 비용이 줄었고 각자 개발에 집중할 수 있었다.

## 본격 개발

### Day 1 - 개발 시작 (10.14)

- **PR 전략**

  1. 최소 1명 이상의 리뷰 필수
  2. 리뷰어는 PR이 딜레이 되지 않도록 우선 응답
  3. 긍정적 피드백 문화 **(**"구웃👍🏼", "이 부분 좋네요!", “할 수 있다!!”)
  4. 학습 공유 (새로 배운 내용은 TIL로 공유)

- **브랜치 전략 & 컴포넌트 개발**

  - Git Flow 전략 수립
    개발 전 이슈를 등록하고, 해당 이슈 번호와 동일한 이름의 브랜치를 생성해 작업을 진행했다. 머지가 완료되면 브랜치를 삭제하는 과정을 반복했다.

    ```
    main (배포)
      └── develop (개발)
           ├── feature/#1 (이슈번호1)
           ├── feature/#2 (이슈번호2)
           └── feature/#3 (이슈번호3)

    ```

  - 커밋 컨벤션
    ```
    feat: 새로운 기능 추가
    fix: 버그 수정
    style: 코드 포맷팅
    refactor: 코드 리팩토링
    test: 테스트 코드
    chore: 빌드, 설정 파일 수정
    ```

> 💡 **배운 점** : 명확한 컨벤션과 Git Flow 전략, 그리고 긍정적인 리뷰 문화가 협업의 효율을 극대화했다.
>
> 특히 브랜치 구조를 통일하니 충돌이 줄고, 코드 리뷰와 머지 과정이 훨씬 체계적으로 진행되었다.

### Day 2-3: 페이지 개발 & 1차 QA (10.15 ~ 10.16)

- **역할 분담**
  주요 컴포넌트 3개를 중심으로 각자 구현하고 싶은 기능을 담당했으며, 작업 완료 순서에 따라 문서 작성과 추가 기능 구현을 분담했다. 이후 테스트용 `mockData.js`를 생성해 기능별 디버깅을 진행했다.

|        | 주요 컴포넌트             | 추가 기능                   |

| 이서정 | 강좌 카드 (Course Card)   | 강좌 등록, 헤더             |
|        |

- **강좌 등록 및 데이터 저장**
  `localStorage`를 마치 API처럼 설계해 데이터 CRUD를 구현했다.

```jsx
export const store = {
  setLocalStorage(key, value) {
    try {
      localStorage.setItem(key, JSON.stringify(value));
      return true;
    } catch (_) {
      return false;
    }
  },

  getLocalStorage(key) {
    try {
      const data = localStorage.getItem(key);
      if (data === null) return null;
      return JSON.parse(data);
    } catch (_) {
      return null;
    }
  },
};
```

> 💡 **배운 점 :** localStorage를 비동기 API처럼 설계하니, 향후 실제 서버 연동 시 코드 수정이 최소화될 것으로 기대된다.

### Day 4-5: 추가 기능 & 2차 QA (10.17 ~ 10.18)

기본 기능이 모두 구현된 후 남은 이틀은 **추가 기능 개발과 품질 점검**, 그리고 **GitHub Pages 배포 환경 구축**에 집중했다.

- **추가 기능 구현**

  - 강좌 수정 및 삭제 기능 : 기존 등록된 데이터를 불러와 수정하고 삭제 시 `alert`메세지를 띄워 실수 방지
  - 검색 기능 강화 : 제목·강사·태그 기반 검색 필터링 구현
  - 정렬 기능 개선 : 최신순, 평점순, 수강인원순 정렬 옵션 추가

- **2차 QA (품질 보강)**
  1차 QA에서의 이슈를 보완하고 2차 QA에서는 예외 상황 중심 테스트를 집중적으로 진행했다.

> 💡 **배운 점** **:** QA는 단순한 오류 탐색이 아니라 서비스 완성도를 높이는 과정이라는 걸 느꼈다.

### Day 6 - 배포**,** UX 개선 및 발표 준비 (10.19)

마지막 날은 발표를 앞두고 팀원 각자가 강점을 살려 역할을 분담했다. 나는 **UX 개선과 브랜딩 작업**을 담당했고 배포 경험이 있는 동료는 **실제 배포 환경 구축**을 맡았다. 또한 팀장은 **전체 발표 흐름과 발표 자료**에 집중해 발표의 완성도를 높였다.

- **GitHub Pages 정적 배포**

시간적 여유가 많지는 않았지만 프로젝트 막바지에 접어들며 회의를 통해 “영상이 아닌 실제 배포된 사이트로 시연하고 싶다”는 모두의 욕심이 생겼고 감사하게도 배포 경험이 있는 동료 덕분에 아직 수업 시간에는 배우지 않았던 **GitHub Pages 배포 과정**을 미리 경험할 수 있었다.

배포 과정에서 **로컬 작업과 배포 환경 간의 경로 차이**로 인해 이슈가 발생했으나 `buildUrl()` 유틸을 활용해 문제를 해결하는 모습을 보았고 이 부분을 모두가 학습 할 수 있도록 공유해주어 상대 경로와 절대 경로의 개념을 이해하고 배울 수 있는 계기가 되었다.

> 💡 **배운 점** **:** 배포 과정은 단순히 결과물을 올리는 단계가 아니라 로컬 환경과 실제 서비스 환경의 차이를 이해하는 학습의 기회였다.

- **UX 개선**

배포를 진행하는 동안 나는 사용자 관점에서 부족한 부분을 다듬고 브랜딩 요소를 추가하는 과정을 진행했다. 좀 더 시각적으로 완성도 높은 경험을 제공하기 위해 **로고 개발, 파비콘, 메인 컬러와 액센트 컬러, 브랜드 슬로건과 핵심 가치 정리**를 중심으로 작업을 진행했고 이 내용을 발표 자료에 반영할 수 있도록 구성안을 작성했다.

![killpo.svg](./images/killpo.svg)

- **발표 준비**

프로젝트 발표를 위해 목표, 일정, 성과, 회고로 구성된 발표 스토리라인을 정리하고 발표 내용을 미리 **스크립트로 작성**하는 과정을 보며 “발표는 이렇게 준비하는 것이구나” 하는 배움의 시간이 되었다. 특히, 스크립트를 작성해두면 **발표 시간을 사전에 조율**할 수 있고**, 전달하고자 하는 의도와 메시지를 명확**하게 다듬는 데 큰 도움이 된다는 것을 느꼈다.

> 💡 **배운 점** **:** 서로의 강점을 존중하며 역할을 나누니 한정된 시간 안에서도 협업 효율과 결과의 완성도를 동시에 높일 수 있었다. 또한, 즉흥적인 발표보다 체계적인 스크립트 기반 준비와 연습이 발표의 완성도와 자신감을 높이는 핵심이라는 걸 알 수 있었다.

### Day 7 - 발표(10.20)

프로젝트의 마지막 날은 그동안의 결과물을 보여주는 발표의 날이다. 우리 팀은 **첫 번째 순서**로 발표를 맡게 되어 긴장감을 감출 수 없었지만, 발표를 담당한 팀장이 최대한 편안하게 발표할 수 있도록 리허설 진행을 함께 점검했다. 또한 발표 직전까지 이슈가 없는지 테스트와 디버깅을 반복하며 발표 후 있을 Q&A를 대비해 내가 맡은 기능과 구현 의도에 대한 답변도 준비했다.

![KakaoTalk_20251023_185133596_03.jpg](./images/KakaoTalk_20251023_185133596_03.jpg)

발표는 정해진 시간 안에 매끄럽게 마무리되었고 Q&A에서도 동료가 명확하고 자신감 있는 답변을 해주며 좋은 인상을 남겼다. 직접 발표나 답변을 하지는 않았지만, 그 모든 과정을 완벽히 해낸 동료들이 정말 자랑스러웠다.

무엇보다 마이크를 잡고 많은 사람들 앞에서 발표하는 것이 쉽지 않은 나에게 이번 경험은 **‘더 좋은 발표 태도와 전달력’을 고민하게 만든 시간**이었다. 동료의 자신감 있는 모습에서 **팀워크와 성장의 진짜 의미**를 배울 수 있었다.

| **함수·이벤트·UI 분리 구조 설계가 어려움**                                  | 초기 구현 시 모든 로직이 한 파일에 집중되어 유지보수성이 낮았음         | 파일 단위 역할을 구분 (`utils/`, `store/`, `components/`)하여 구조 정리, 추후 확장 용이하게 개선  |
| **입력 폼 항목이 많아 UX 중심의 로직 구성에 어려움**                        | 필수 필드 검증, alert 메시지 처리, 입력 흐름 제어 등 복합적인 검증 필요 | 입력 검증 로직을 단계별로 분리하고 alert 메시지 단순화, 추후 사용자 경험 중심의 UI 개선 여지 확보 |

> 💡 **배운 점** : 프로젝트에서 가장 큰 성장은 단순히 오류를 해결한 것이 아니라 문제의 원인을 구조적으로 파악하고 재발하지 않도록 개선한 과정에서 이루어졌다.
>
> 특히, 카테고리 선택 기능을 구현하면서는 **데이터 구조 설계와 UI 로직의 분리**가 얼마나 중요한지를 깨달았다. 기능이 복잡해질수록 코드의 책임을 명확히 분리해야 유지보수가 쉬워진다는 것을 직접 경험했다. 또한 **문제를 발견하고 즉시 동료와 공유**하며 해결 방향을 논의하는 과정에서 \*\*\*\*협업의 효율성도 향상되었다.
